# NFTEnex No-Budget E-E-A-T Framework

This framework defines how NFTEnex should publish editorial comparison content when the team does not have budget for full live transactional testing.

The goal is not to fake hands-on use. The goal is to publish transparent, well-observed, documentation-backed comparisons that still earn trust.

## Core Rule

Do not claim direct product use where none happened.

If the team only reviewed:

- live public product surfaces
- official documentation
- help-center pages
- visible workflow cues

then say exactly that.

## Publishing Position

These pages should be framed as:

- editorial comparisons
- public-surface reviews
- documentation-backed guides
- workflow-based evaluations

Do not frame them as:

- fully tested hands-on reviews
- end-to-end live transaction reports
- product reviews based on funded onchain execution

unless the article actually includes that evidence.

## Required Top-of-Article Structure

Every article should include these four blocks near the top in this order:

1. Author and update box
2. Trust block
3. Methodology block
4. Limitations block

## Exact Author and Update Box

Use this exact format:

```md
> Reviewed by NFTEnex Editorial Team
> Last reviewed: 2026-07-13
> Review type: No-budget editorial comparison
```

## Exact Trust Block

Use this exact format:

```md
> Why you can trust this guide
>
> This guide is based on live public product surfaces and official references reviewed on 2026-07-13. We directly checked the public positioning, visible workflow framing, and documentation shown in this article. We do not present unverified logged-in behavior, live checkout results, or completed onchain actions as first-hand use unless they were actually completed and documented.
```

## Exact Methodology Block

Use this exact format:

```md
> Methodology
>
> We compared each option using live public product surfaces, official documentation, and visible workflow cues captured at review time. In this version, the ranking prioritizes clarity, workflow posture, and fit for different user types over private dashboard claims we could not verify directly.
```

## Exact Limitations Block

Use this exact format:

```md
> Limitations
>
> This is a no-budget editorial review, not a fully funded end-to-end product test. Where a conclusion would require a live transaction, paid plan, logged-in dashboard, or wallet-funded workflow, we treat that as a limitation and avoid overstating direct experience.
```

## First-Hand Language Rules

### Safe phrases

- `we reviewed`
- `we checked`
- `we noticed`
- `what stood out immediately was`
- `from the public surface we reviewed`
- `based on what we could verify directly`

### Unsafe phrases unless fully true

- `we tested everything`
- `we used this in production`
- `we completed the workflow` without actual proof
- `we confirmed live results` without actual evidence

## Evidence Hierarchy

When money is limited, use this hierarchy:

1. Original screenshots from live public pages
2. Official documentation and help-center pages
3. Official pricing or policy pages
4. Visible workflow structure on public-facing product surfaces
5. Clear editorial judgment about tradeoffs

Do not rely on:

- anonymous third-party summaries
- vendor promo copy alone
- old screenshots from other sites
- social posts as primary proof of product behavior

## Required Visual Workflow

Each image block should include:

- `Featured Image`
- `Screenshot 1`
- `Screenshot 2`
- `File`
- `Alt text`
- `Caption`

Keep captions:

- editorial
- natural
- SEO-clean
- specific to the article topic

## What This Framework Still Proves

Even without live funded tests, this framework still proves:

- the content is current
- the team directly reviewed something real
- the team understands the product posture
- the team can explain tradeoffs
- the team is honest about evidence boundaries

That is still stronger than generic listicle content that pretends certainty without proof.

## Update Rule

Recheck volatile claims before publish and during refresh cycles:

- fees
- chain support
- live availability
- onboarding or payment claims
- marketplace or API positioning

If one of those changes, update:

- the sentence in-body
- the last reviewed date
- any screenshot that no longer reflects current product posture
